# 理想投资 Agent：需求摘要与仓库对照（路线图）

本文档将对话中归纳的「用户画像、能力期望、功能分层」与 **本仓库 Stock_Analysis 现状** 对照，用于排期与边界说明。  
**合规口径**：仅作产品与技术分析方向的说明；不构成投资建议，不承诺任何回测或模拟结果。

---

## 一、用户画像与痛点（产品侧假设）

| 维度 | 内容 |
|------|------|
| 投资风格 | 偏长期、基本面；持有大型科技龙头；也关注周期品（如利率逻辑下的贵金属等）。 |
| 交易习惯 | 很少日内 / T+0，以持有为主。 |
| 主要痛点 | 有效信息少、噪声多；复杂新闻对资产的影响难判断；短期判断压力大。 |

**与仓库关系**：本仓库是**工具与数据管道**，不内置用户画像；痛点中「信息线索」可通过研报检索等**弱补强**，「新闻影响建模」当前未做。

---

## 二、对「理想投资 Agent」的能力期望（归纳）

### 2.1 基本面

能分析公司基本面，并用于判断中长期价值。

**仓库现状**：**弱**。无财报/一致预期/估值引擎；`intel` + 研报客可提供**检索与摘要线索**（`--with-research`），**不能替代**基本面模型或投研结论。

### 2.2 价值 vs 价格

把「公司核心价值」与「当前市价」对照，辅助是否高估/低估的**框架**（非喊单）。

**仓库现状**：**基本未做**。当前输出以 **K 线结构、威科夫背景、123 计划价、Fib 区间** 为主，属于**技术向**「价」与「位」的参考，不是基本面估值框架。

### 2.3 风险与组合

结合持仓、总资金、现金流做动态分析；支持预设回撤容忍；输出「风险值」类指标；可用蒙特卡洛等刻画**极端情景**（表述为风险模拟，**不承诺**结果）。

**仓库现状**：**基本未做**。无组合模块、无账户级参数、无蒙特卡洛或组合在险价值类引擎。单标的战术止损/时间止损见 `analysis/kline_metrics.py` 与台账字段，**不等于**组合风险。

### 2.4 资金与仓位

在总净值与目标仓位比例上给建议框架；结合价、量等，将买卖资金拆到不同价位（分批加减仓思路）。

**仓库现状**：**部分**。L3 层面有入场区、止损、止盈等级等**价位参考**（见 `cli/stock_analysis.py` 台账与 `analysis/kline_metrics.py`）；**无**净值、目标仓位、资金拆单与预算约束的闭环。

## 三、功能分层（四级依赖）

层级名称（意译）与职责如下；依赖关系：**L1 定边界 → L2 分析 → L3 在边界内出价位方案 → L4 持续记账与回测输入**。

| 层级 | 名称 | 职责 |
|------|------|------|
| L1 | 战略与风险边界 | 总策略、风险边界（如最大止损比例、总敞口等硬约束）。 |
| L2 | 分析 | 基本面与中长期价值判断（承接「看未来价值」）。 |
| L3 | 战术执行 | 具体价位参考：入场、止盈止损、结构位与计划（可与技术结构类比）。 |
| L4 | 数据与台账 | 记录与跟踪：持仓均价、历史成交、统计与复盘输入。 |

### 与本仓库的覆盖关系（摘要）

| 层级 | 仓库覆盖程度 | 说明 |
|------|----------------|------|
| L1 | **未覆盖** | 需新增组合/账户配置与风控边界模块（若做）。 |
| L2 | **弱** | 研报线索可辅助「找材料」，不是深度基本面引擎。 |
| L3 | **已覆盖（偏技术）** | `analysis/kline_metrics.py`（威科夫 + 123、Fib、多周期 `mtf_v1`、结构过滤 `structure_filters_v1`、时间止损元数据等）；`cli/stock_analysis.py` 生成简报与 `ai_overview.json`。 |
| L4 | **已覆盖** | PostgreSQL `journal_*`、`analysis/ledger_stats.py`（周/月统计、按 status / `wyckoff_bias` 分层、时间止损过期挂单提示等）；可读 Markdown 快照。 |

## 四、延伸问题：是否自动买卖（产品红线）

**结论（建议写进需求）**：自动下单属于**执行层**，涉及实盘接口、合规、适当性、风控审批与人工确认；宜列为**阶段目标 / 非首期范围**，并在需求中单列**红线**（例如：默认禁止自动实盘、仅分析与模拟、任何执行须显式人工确认）。

**本仓库现状**：**未实现自动交易**；当前以拉行情、算指标、写报告与台账为主，与「执行层」分离，有利于安全与合规边界清晰。
